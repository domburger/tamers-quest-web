Tamers Quest — Game Logic

A reference for verifying that the game's rules and formulas are correct. CURRENT = how the code works today (the multiplayer extraction loop is live). PLANNED = designed but not yet built (e.g. AI content gen). OPEN = awaiting a decision from you.

This doc mirrors the code. When a rule here disagrees with how you want the game to behave, that's a logic bug to flag — note it and we fix the code (or this spec). Companion: docs/IMPLEMENTATION_PLAN.md.

Overview

Tamers Quest is a real-time online multiplayer extraction game with monster taming (think Dark and Darker), live at tamersquest.com: you bring a team of monsters into a shared procedural map with up to 16 players, fight wild monsters, and try to extract through a portal before the closing safe zone ends your run. PvP is live — players duel on contact or by landing a spirit chain, and the winner loots the loser's team. Combat is turn-based, AI-resolved with a deterministic engine fallback.

Game Loop

  1. Menu → pick/create a character. CURRENT
  2. Base inventory — the player owns a roster of monsters: a team of 4 active + a vault of up to 100. CURRENT
  3. Queue into a round — matchmaking gathers up to 16 players (or starts after a countdown with ≥ min). CURRENT
  4. Spawn on a procedurally-generated map (DLA caves + Voronoi biomes), at a server-assigned walkable point from the seed. CURRENT
  5. Explore top-down (WASD / joystick), server-side tile collision. CURRENT
  6. Encounter wild monsters — some visible, some hidden (ambush). Walk into one (it acts first) or throw a spirit chain to engage with initiative → instanced turn-based combat / taming. CURRENT PvE and PvP (rivals duel on contact or by chain) CURRENT
  7. Extract via a portal (keep your gains) or lose your run team on defeat. CURRENT
  8. Results return you to the menu; profile saved server-side. CURRENT (in-memory; durable Postgres is coded but awaiting DB provisioning)

Multiplayer Model CURRENT

Online multiplayer is live at tamersquest.com — one combined Node service serves the client and runs the authoritative WebSocket game on the same origin.

  • Up to 16 players share one round on one map instance; multiple rounds run concurrently.
  • Authoritative server owns all state (positions, monsters, combat, zone). Clients send inputs & render snapshots — never trusted (see Anti-cheat).
  • Map by seed: the server picks a seed and sends only that; clients regenerate the identical map deterministically (seeded RNG).
  • Sessions: anonymous play with an optional nickname; the server issues an opaque token the client stores to resume the same profile on reconnect (Google/Discord auth later).
  • Visibility: the server filters each snapshot to an area of interest — some monsters spawn hidden and only reveal at close range (ambush).

Movement & Sprint CURRENT

Top-down movement at BASE_SPEED (200px/s), with per-axis tile collision (slide along walls). Hold Shift to sprint — move faster while stamina lasts. Server-authoritative online (the sprint flag rides the move input; the server integrates speed + stamina and broadcasts it), local in single-player; both share the pure helpers in src/engine/movement.js.

Monsters

Monsters are AI-generated as data (stats, scaling, attacks, element, lore) and procedurally rendered as canvas shapes (no PNGs). 103 exist today.

Each MonsterType carries: typeName, element, rarity (1–5), size, base stats + two scaling coefficients per stat, four attacks, a passive & active effect, and a description.

Procedural visuals CURRENT: a seeded generator (src/systems/spritegen.js) draws a creature deterministically from the monster's name — body shape, eyes, rarity-driven spots, and element features (fire spikes, water fins, nature leaves, dark horns, light rays, neutral ears), colored by an element palette.

Stats & Scaling CURRENT

Seven stats: health, strength, defense, speed, power, energy, luck.

A stat at a given level:

stat(level) = floor( base + scaling1 * level ^ scaling2 )

Each stat has its own base, scaling1, and scaling2 from the monster's data row. Example (Flamepaw Lynx speed: base 95, s1 1.5, s2 1.2) at level 3: floor(95 + 1.5 * 3^1.2) ≈ floor(95 + 5.6) = 100.

Combat CURRENT

Instanced & turn-based: when you walk within range of a wild monster the server starts a private duel (others keep moving). Each turn is AI-resolved on the server (OpenAI, the core feature, decision Q3); the deterministic engine (src/engine/combat.js) is the automatic fallback when no API key is set or the call errors, and the offline baseline. Catch is always deterministic for now.

The deterministic rules below define the fallback and the baseline the AI is expected to roughly follow.

Turn order

Higher speed acts first; ties favor the player.

Damage formula

physical = attacker.strength * (attack.damage / 100)
           - defender.defense * (1 - attack.penetration)
elemental = attacker.power * attack.elementalDiffusion
raw      = max(1, floor(physical + elemental))
// crit (see below) may multiply here
final    = floor(raw * elementMultiplier)   // min 1 per hit

Accuracy

hit if  rand(0..100) <= attack.accuracy * 100 + attacker.luck / 2

Critical hits

crit if rand(0..100) <= attack.critChance * 100 + attacker.luck / 4
on crit: damage *= attack.critMultiplier

Energy

Each attack costs energyCost. If a monster can't afford its action, it skips. Energy does not regenerate during a fight. Between encounters (decision Q8) every living team monster regains 50% of max energy (capped) at the start of each new fight, so a depleted team isn't stuck skipping forever.

Updated 2026-06-06: the deterministic engine (src/engine/combat.js) now implements the full spec — speed-based turn order, crit rolls for both sides (old bug fixed), and status ticks + application. systems/combat.js delegates to it. Verified in Node.

Status Effects CURRENT

Only one status at a time; a new status replaces the old. Applied via attack.statusChance. Now enforced by the deterministic engine (synonyms normalized: Stunned→Stun, Frozen→Freeze, Poisoned→Poison).

Resolved (Q7 — taxonomy shelved): the attack data names ~50 distinct status labels (Bleed, Blind, Confusion, Fear, Paralysis…) plus buffs (Heal, Shielded, Reflect). Rather than hand-code a taxonomy, the AI resolver interprets statuses freely during live fights. The four mechanics above are the deterministic fallback's canonical set for offline play only.

Spirit Chains CURRENT

Spirit Chains are throwable items — the core verb for both engaging and capturing. You aim and throw a chain on the overworld; it flies in your facing direction and can miss. Definitions live in public/assets/data/spiritchains.json; capture/throw math is the shared pure module src/engine/spiritchains.js; tunables sit in GAME.SPIRIT_CHAIN.

  • Engage by throw: landing a chain on a monster starts combat and gives the thrower the first turn (initiative); walking into a monster still starts combat, but the monster acts first. Throwing at a player engages a PvP duel and the thrower gets the first-turn initiative edge (applied on turn 1, then normal). CURRENT
  • Capture stays turn-based: a map throw only starts the fight; the in-combat catch is what captures, now scaled by the chain's tier (see Taming). CURRENT
  • Two resources per chain: throwCount (overworld throws — spent on every throw, hit or miss) and durability (capture charges — spent only on a successful capture; the chain is consumed when charges hit 0). CURRENT
  • Controls: Q throws along your facing; [ / ] cycle the equipped chain. Owned chains are also viewable/equippable in the Inventory → Spirit Chains tab (tier · throws · charges · equipped). CURRENT

Acquisition

  • Starter: every profile is granted a tier-1 Frayed Chain (and back-filled on older saves). CURRENT
  • Loot chests: 10 chests/round spawn against walls (server-authoritative, seeded), each holding 1–2 randomized chains weighted by dropWeight (common low tiers, rare specials). Walk within 40px to open; a duplicate refills that chain's throws + charges. Chests blip on the minimap only within 420px — you explore to find them. CURRENT
  • Gold shop — spend run-earned gold between runs (single-player lobby and the online lobby, both wired to the server-authoritative buy). Prices: T1 25g · T2 70g · T3 160g · T4 380g · T5 850g · specials 1300–1600g. Buying banks the chain permanently (a duplicate refills its throws + charges); the shop is locked while you're in a run. CURRENT
  • Crafting — spend Spirit Essence (a material earned in runs: +2 per monster defeated, +3 per chest) to upgrade an owned base-tier chain to the next tier (consumes the lower chain). Cost = 40 × from-tier essence (T1→T2 = 40 … T4→T5 = 160). Done in the Inventory → Spirit Chains tab (SP) and the online Spirit Shop (MP). CURRENT
Extraction stakes: chains opened from chests this run are provisional (runFound) — banked on extract, lost on death/timeout (your starter and previously-banked chains are always safe; a death never leaves you chainless). A refill of an already-banked chain is not at risk. CURRENT

Taming (Catch) CURRENT

Catch with a spirit chain during combat. Base chance by enemy HP %, then scaled by the equipped chain's capture multiplier (and gated by its max rarity):

Enemy HPBase catch chance
< 25%70%
25–50%40%
50–75%20%
> 75%5%

Modifiers, in order: +15% if the enemy has a status effect → × the chain's capture multiplier (0.50× tier 1 … 1.60× tier 5) → clamped to 95% (a guaranteed chain bypasses the clamp at ≤25% HP) → auto-fail if the enemy's rarity exceeds the chain's max rarity. The enemy still attacks during the attempt, except on a first-turn player-initiated catch (a thrown-chain engage), which skips the retaliation. On success the monster joins the team (or the vault if the team is full of 4) and a chain charge is consumed.

Progression CURRENT

  • XP per level: 100. On reaching 100, level +1 and the carry-over resets; HP/energy refill to the new level's max.
  • Defeat a monster: 20 + level*10 XP and 4 + level*2 gold.
  • Catch a monster: 30 + level*15 XP.
  • Extract: +30 gold run-completion bonus.
  • Gold: persists across runs (kept on death) and is spent in the spirit shop.
  • Spirit Essence: +2 per defeat, +3 per chest; persists across runs; spent to craft/upgrade chains.
  • Team: 4 active slots. Vault: up to 100.

Map Generation CURRENT

Three-stage pipeline, ported from the original Java game:

  1. DLA cave carving — Diffusion-Limited Aggregation carves walkable space until ~35% of the map is open.
  2. Voronoi biomes — biome seeds scatter; each tile takes its nearest seed's biome (with smoothing passes).
  3. Tile placement — for each walkable cell, score candidate tiles from that biome by matching color profiles to neighbors, trying 4 rotations (ROT_MAP), and place the best match.

The online view draws culled, biome-colored tiles and a corner minimap; main-view camera zoom is a remaining polish item.

Rendering & HUD CURRENT

Engine: Kaboom.js on a single WebGL canvas (chosen over DOM/SVG for performance at 400×400 + 16 players). All art is procedural canvas generation — no PNG assets. The player is an animated character drawn from Kaboom primitives (idle bob + walk cycle); monsters/tiles are procedural sprites/rects.

The online HUD includes:

  • Minimap / radar (top-right): the shrinking safe zone, extraction portals, nearby monsters/players, and your position over faint downsampled terrain.
  • Team HP bars (top-left): live per-monster HP from the snapshot, so storm/combat damage to your reserves is visible.
  • Danger warning: a pulsing red border + "OUTSIDE SAFE ZONE" when you're in the storm.
  • Combat overlay: element dots, color-coded HP + energy bars, status chips, and element-tinted attack buttons showing energy cost.
  • Mobile: virtual joystick, tappable combat buttons, an HTML nickname input (real soft keyboard), and PWA install (manifest + icons + service worker).
