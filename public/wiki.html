A reference for verifying that the game's rules and formulas are correct. CURRENT = how the code works today. PLANNED = the multiplayer-extraction vision. OPEN = not yet decided.

This doc mirrors the code. When a rule here disagrees with how you want the game to behave, that's a logic bug to flag — note it and we fix the code (or this spec). Companion: docs/IMPLEMENTATION_PLAN.md.

Overview

Tamers Quest is a procedurally-generated dungeon crawler with monster taming and turn-based combat. The target design is a real-time online multiplayer extraction game (think Dark and Darker): you bring a team of monsters into a shared procedural map with up to 16 players, fight wild monsters and rivals, and try to extract through a portal before the closing zone or other players end your run.

Game Loop

  1. Menu → pick/create a character. CURRENT
  2. Base inventory — the player owns a roster of monsters: a team of 4 active + a vault of up to 100. CURRENT
  3. Queue into a round — matchmaking gathers up to 16 players. PLANNED
  4. Spawn on a procedurally-generated map (DLA caves + Voronoi biomes), at a server-assigned point. CURRENT gen / PLANNED networked spawn
  5. Explore top-down (WASD), tile collision & speed modifiers. CURRENT
  6. Encounter monsters (some visible, some hidden) and rival players → turn-based combat / taming. CURRENT PvE / PLANNED PvP & hidden spawns
  7. Extract via a portal (keep your gains) or lose your run team on defeat. CURRENT
  8. Results persist to the account; return to menu. CURRENT local / PLANNED server

Multiplayer Model PLANNED

  • Up to 16 players share one round on one map instance.
  • Authoritative server owns all state (positions, monsters, combat, loot, zone). Clients send inputs & render snapshots — they are never trusted (it's PvP with loot).
  • Map by seed: the server generates the map from a seed; clients reproduce it deterministically (requires seeded RNG — see plan P0-T2).
  • Visibility: the server decides which monsters/players each client can see (some monsters hidden until close).
The current code is single-player and stores everything in localStorage with Math.random(). Multiplayer is a re-architecture around a server — tracked as P0–P2 in the plan.

Monsters

Monsters are AI-generated as data (stats, scaling, attacks, element, lore) and procedurally rendered as canvas shapes (no PNGs). 103 exist today.

Each MonsterType carries: typeName, element, rarity (1–5), size, base stats + two scaling coefficients per stat, four attacks, a passive & active effect, and a description.

Procedural visuals CURRENT: a seeded generator (src/systems/spritegen.js) draws a creature deterministically from the monster's name — body shape, eyes, rarity-driven spots, and element features (fire spikes, water fins, nature leaves, dark horns, light rays, neutral ears), colored by an element palette.

Stats & Scaling CURRENT

Seven stats: health, strength, defense, speed, power, energy, luck.

A stat at a given level:

stat(level) = floor( base + scaling1 * level ^ scaling2 )

Each stat has its own base, scaling1, and scaling2 from the monster's data row. Example (Flamepaw Lynx speed: base 95, s1 1.5, s2 1.2) at level 3: floor(95 + 1.5 * 3^1.2) ≈ floor(95 + 5.6) = 100.

Combat CURRENT AI in MP: OPEN

Turn-based. Each turn both monsters pick an action; the engine resolves it. Two engines exist with identical rules: an AI engine (GPT-4o) and a deterministic fallback (used when no API key, and the basis for the future server-authoritative resolver).

Turn order

Higher speed acts first; ties favor the player.

Damage formula

physical = attacker.strength * (attack.damage / 100)
           - defender.defense * (1 - attack.penetration)
elemental = attacker.power * attack.elementalDiffusion
raw      = max(1, floor(physical + elemental))
// crit (see below) may multiply here
final    = floor(raw * elementMultiplier)   // min 1 per hit

Accuracy

hit if  rand(0..100) <= attack.accuracy * 100 + attacker.luck / 2

Critical hits

crit if rand(0..100) <= attack.critChance * 100 + attacker.luck / 4
on crit: damage *= attack.critMultiplier

Energy

Each attack costs energyCost. If a monster can't afford its action, it skips. Energy does not regenerate during a fight.

Note (current code): in the deterministic fallback, the player's crit roll is applied but the enemy's crit roll is not; status-effect application (burn/poison/etc.) is described in the AI prompt but the fallback engine does not apply or tick statuses. Flag if this should be unified before the server resolver is built.

Taming (Catch) CURRENT

Instead of attacking, throw a catch. Base chance by enemy HP %:

Enemy HPBase catch chance
< 25%70%
25–50%40%
50–75%20%
> 75%5%

Modifiers: +15% if the enemy has a status effect. (The AI prompt also mentions +5% per player level above enemy and a rarity penalty; the deterministic fallback applies only the status bonus.) The enemy still attacks during the catch attempt. On success the monster joins the team (or the vault if the team is full of 4).

Progression CURRENT

  • XP per level: 100. On reaching 100, level +1 and the carry-over resets; HP/energy refill to the new level's max.
  • Defeat a monster: 20 + level*10 XP.
  • Catch a monster: 30 + level*15 XP.
  • Team: 4 active slots. Vault: up to 100.

Map Generation CURRENT

Three-stage pipeline, ported from the original Java game:

  1. DLA cave carving — Diffusion-Limited Aggregation carves walkable space until ~35% of the map is open.
  2. Voronoi biomes — biome seeds scatter; each tile takes its nearest seed's biome (with smoothing passes).
  3. Tile placement — for each walkable cell, score candidate tiles from that biome by matching color profiles to neighbors, trying 4 rotations (ROT_MAP), and place the best match.

Rendering CURRENT

Engine: Kaboom.js on a single WebGL canvas (chosen over DOM/SVG for performance at 400×400 + 16 players). All art is procedural canvas generation — no PNG assets. Sprites are registered under stable names so scene code is render-agnostic.
